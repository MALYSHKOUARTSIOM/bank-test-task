# Bank Test Task (REST API)

REST API для управления клиентами и их платежными картами. 

Проект построен на базе Spring Boot и представляет собой структурированное решение с разделением слоев (Controller-Service-Repository), валидацией данных, динамическим поиском, системой миграций базы данных, гибридной авторизацией (Basic + JWT) и автотестами.

---

## 🛠 Технологический стек
* **Java 17**
* **Spring Boot 3.3.1** (стабильная версия)
* **Spring Data JPA** & Hibernate
* **PostgreSQL** (основная СУБД)
* **H2 Database** (встроенная БД для выполнения тестов в изолированном окружении)
* **Liquibase** (управление миграциями и структурой БД)
* **Spring Security & JJWT** (защита API)
* **Springdoc OpenAPI 2.6.0** (интерактивный Swagger UI)
* **Lombok** & MapStruct (маппинг DTO)
* **JUnit 5 & Mockito & MockMvc** (автотесты)

## 🚀 Как запустить приложение

### 1. Подготовка базы данных
Приложению требуется СУБД PostgreSQL.
1. Создайте базу данных с именем **`bank_db`** (например, через pgAdmin или консоль psql):
   ```sql
   CREATE DATABASE bank_db;
   ```
2. Настройки подключения по умолчанию (`src/main/resources/application.properties`):
   * URL: `jdbc:postgresql://localhost:5432/bank_db`
   * Логин: `postgres`
   * Пароль по умолчанию: `1234`
   
   *(Если ваши учетные данные отличаются, вы можете передать их через переменные окружения `DB_USERNAME` и `DB_PASSWORD` при запуске).*

### 2. Запуск проекта
Запустите приложение из корня проекта с помощью Maven Wrapper:
```bash
# Для Windows (PowerShell):
$env:DB_PASSWORD="ваш_пароль"; .\mvnw spring-boot:run

# Для Linux / macOS / Git Bash:
DB_PASSWORD="ваш_пароль" ./mvnw spring-boot:run
```

При запуске **Liquibase** автоматически применит все миграции: создаст таблицы `client`, `payment_card` и наполнит БД тестовыми данными.

## 📖 Документация API (Swagger UI)
После запуска приложения документация API доступна по адресу:
👉 **[http://localhost:8080/swagger-ui/index.html](http://localhost:8080/swagger-ui/index.html)**

### Схема авторизации в Swagger:
В приложении настроена строгая раздельная безопасность:
1. **Получение токена (`POST /api/auth/token`):**
   * Требует прохождения **Basic Auth**.
   * Нажмите зеленую кнопку **Authorize** в правом верхнем углу Swagger.
   * В секции **`basicAuth`** введите учетные данные по умолчанию:
     * **Логин:** `admin`
     * **Пароль:** `admin`
   * Нажмите Authorize, закройте окно. Вызовите метод `POST /api/auth/token` (нажав *Try it out* -> *Execute*).
   * Скопируйте полученный JWT-токен из тела ответа.
2. **Доступ к остальным эндпоинтам (клиенты, карты):**
   * Требует **только JWT-токен**. Вход по логину/паролю на этих методах заблокирован.
   * Снова нажмите кнопку **Authorize** вверху страницы.
   * Нажмите *Logout* напротив `basicAuth` (чтобы стереть старые учетные данные).
   * Вставьте скопированный JWT-токен в поле **`bearerAuth`** и нажмите **Authorize**.
   * Теперь вы можете вызывать любые бизнес-методы (создание клиентов, выпуск карт, поиск).

## 💡 Основные фичи реализации
* **Динамический поиск клиентов (`GET /api/clients/search`):** Реализован с помощью JPA Specifications. Выполняет `LEFT JOIN FETCH` таблиц клиентов и карт, защищая от проблемы N+1 запросов, поддерживает регистронезависимый поиск (`LOWER`) и поиск по частичному совпадению (`LIKE`).
* **Логика изменения статуса лояльности:** Статус клиента лояльности (`Classic -> 0, Gold -> 1, Platinum -> 2`) пересчитывается автоматически при выпуске карты. Реализовано правило «только апгрейд» — клиент не теряет премиальный статус при выпуске карты младшего ранга.
* **Глобальная обработка ошибок:** Все ошибки валидации форм (например, невалидный email или телефон) перехватываются `@RestControllerAdvice` и возвращаются клиенту в виде структурированного JSON-ответа со статусом `400 Bad Request`.
